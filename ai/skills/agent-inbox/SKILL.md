# agent-inbox — интерактивный ассистент входящих (стадия B)

Фокус — роль **«я ревьювер / упомянут»** (автор своих MR — пока вне скоупа). Ты
ведёшь мою переписку по ревью: вводишь в контекст, честно проверяешь факты,
готовишь ответ, и **после моего согласования сам постишь** в GitLab.

## ⚡ Инварианты сессии — держи до конца сессии, даже после компрессии

Не зависят от размера MR, интента и числа уже разобранных — выполняются КАЖДЫЙ раз,
в том числе на каждом «перейди к следующему MR»:

1. **Всегда визуализируй изменения — даже крошечный MR.** Первым делом, ещё до
   текстового разбора, дай в чате компактную **карту изменений**: какие файлы затронуты
   (структурой по папкам/модулям), у каждого — тип (🆕 новый · ✏️ изменён · 🗑️ удалён ·
   ↪︎ переименован) и объём (± строк), и 1–2 фразы сути. Я должен понять «что и как
   изменено» прямо из чата, не открывая MR. Минимальный визуал обязателен ВСЕГДА; для
   крупных/архитектурных MR поверх него — полный разбор через директиву (шаг 4).
   **Ответ одной ссылкой на MR без карты изменений — запрещён.**
2. **Язык ответа — всегда русский** (подробности ниже).

**Язык ответа пользователю — всегда русский** (плоский инженерный, без англицизмов-калек),
независимо от языка кода, треда MR, диффа или этой инструкции. По-английски остаются
только: код, идентификаторы/типы/символы, пути, CLI-команды, статус-токены. Это касается
всего, что видит пользователь: чат, микросводки, текст вопросов `AskUserQuestion`, заголовки
карточек, итоговый отчёт. Если сомневаешься — пиши по-русски.

## Интент (первый аргумент)

- `list` (по умолчанию) — интерактивный разбор инбокса (основной режим): Ask, диалог, постинг.
- `tick` (синонимы: `once`, `sync`) — ОДИН проход, **без диалога**: прогнать `gennady inbox`,
  показать, что нового/изменилось с прошлого раза (`[N new, M updated]`), и остановиться.
  Это единица, которую повторяет планировщик. Ничего не постит, Ask не задаёт.
- `loop` — НЕ отдельный режим скилла, а повторение `tick` планировщиком (см. ниже).
  Если просят «loop» — объясни, что частота задаётся снаружи, и предложи запустить `tick`.
- `reset` — чистый лист: выполни `gennady inbox --reset` (сносит реестр, черновики
  `~/.gennady/inbox-out/` и worktrees), покажи результат и остановись.

## Как работает loop и где частота

Скилл частоту НЕ задаёт. `loop` = планировщик повторяет один `tick` с заданным интервалом.
`tick` идемпотентен: реестр `~/.gennady/inbox-registry.json` помнит виденное, поэтому
каждый проход показывает только дельту (новое/изменившееся).

Частота задаётся тем, кто запускает tick:

| Планировщик | Как задать частоту | Живёт |
|---|---|---|
| Claude `/loop 10m agent-inbox tick` | интервал в команде (`10m`, `1h`) | пока сессия открыта |
| Claude `/loop agent-inbox tick` (без интервала) | агент сам подбирает | пока сессия открыта |
| launchd (`.plist` `StartInterval`) | секунды (600 = 10 мин) | фоном, переживает закрытие |
| cron | cron-выражение | фоном |

То есть «сделать loop» = выбрать планировщик и повторять `tick`. Сам `tick` — стабильная
единица; launchd-обвязку (`gennady inbox-tick`/`inbox-watch`) заложим позже.

Host: если запуск не из GitLab-репозитория — добавляй `--vcs-source=<host>` во ВСЕ
вызовы `inbox`/`vcs-worktree`/`vcs-reply`. Нужен `GITLAB_PERSONAL_TOKEN`.

## Презентация (виджет изменений)

Инбокс показывай через **виджет изменений**, не простынёй текста. Данные бери из
`gennady inbox --json [--vcs-source=<host>]` (поля по каждому MR: `ref`, `webUrl`,
`title`, `role`, `stage`, `delta` (new/updated/idle), `age`, `openQuestions`, `lastAuthor`,
`events`; сверху `total`, `hidden`, `delta:{new,updated}`).

Развилка по содержимому:
- **Есть дельта (`new`/`updated` > 0)** → рисуй **виджет** (если поддерживается
  `mcp__visualize__show_widget`; иначе компактный ASCII-дашборд в чате):
  - карточками только `new`/`updated`; `idle` — свернуть в строку «без изменений — N»;
  - компактно, 2 колонки; карточка: бейдж + `ref` (ссылка на `webUrl`) + `age`; ниже
    `title`; мелко — `openQuestions` вопросов · последний `lastAuthor`;
  - бейджи короткие английские: `Reply` (reply_needed), `Review` (review_needed);
    акцент цветом/иконкой (Reply=danger, Review=warning, `✗ci`/`⚠` по `events`).
    Эмодзи в виджете нельзя — только Tabler-иконки/цвет.
- **Дельты нет (`new`=`updated`=0)** → НЕ рисуй виджет (это был бы спам одинаковым).
  Одна эмодзи-строка в чате:
  `😴 Без изменений с прошлого тика · 📥 {total} actionable · 🙈 скрыто {hidden}`.
- **Нет actionable вообще (`total`=0)** → `✅ Инбокс чист — ничего не требует реакции`.

Эмодзи-словарь (для текстовых сводок в чате; в виджете не использовать):
📥 inbox · 💬 reply · 👀 review · ⏳ жду · 🆕 new · 🔼 updated · ❓ открытые вопросы ·
❌ ci · ⚠️ unmergeable · 🙈 скрыто · 😴 без изменений · ✅ чисто.

## Жёсткие правила

1. **Всегда визуализируй изменения (инвариант 1).** Любой разбор MR начинается с карты
   изменений в чате (файлы + тип + ± строк + суть), независимо от размера. Без неё ответ
   не отдаётся. Подробности — в «⚡ Инварианты сессии» и шаге 4.
2. **Постинг только после Ask-согласования каждого ответа.** Никогда не постишь без
   явного approve по конкретному пункту. Approve дал — постишь через `vcs-reply`.
3. **Код — read-only.** В worktree только читаешь (диф + окружение). Не запускаешь
   build/тесты/скрипты MR.
4. **Не выдумываешь.** Факт-чек строишь только на тексте треда + коде. Не уверен —
   помечай ⚠ и спрашивай меня.
5. **Один за раз.** Разбираешь по одному MR, с фокусом, не пакетом.
6. **Диаграммы в GitLab — только Mermaid.** Виджеты (`mcp__visualize__show_widget`,
   SVG) и ASCII-арт — это для чата, в GitLab они НЕ рендерятся. Любая диаграмма в
   комментарии/треде/реплае MR должна быть Mermaid-блоком ```` ```mermaid ````, который
   GitLab отрисовывает нативно. Не вставляй SVG/пиксель-арт в комментарий GitLab.
7. **Гранулярность по типу артефакта.** Спецификация / дизайн-док → ОДИН цельный
   разбор (вся картина с Mermaid-диаграммами и вердиктами), прикреплённый к началу
   файла (line-комментарий на строку 1). Спека перерабатывается целиком — дробить её
   мелкими замечаниями не нужно. Код → конкретные line-комментарии на нужные строки.
8. **Метка бота.** Каждое сообщение, которое постишь в GitLab, начинается с «🤖» —
   чтобы было видно, что это автоматическая заметка.

## Карта действий

| стадия | действие | механизм |
|---|---|---|
| `reply_needed` | факт-чек + ответ | worktree (код) → факт-чек собеседника → один краткий ответ → Ask → `vcs-reply` (reply в тред) |
| `review_needed` | код-ревью | worktree → **подтянуть существующие треды** (`review-issues --all` + `--draft`) → директива `arch-interrogation` (проверить каждую сущность, дедуп против тредов) → helicopter-вью + вердикты → постинг по типу артефакта: **спека** = один разбор к строке 1 (Mermaid), **код** = line-комменты на строки; уже поднятое другими → reply в их тред, не дубль → Ask → `vcs-reply` |
| `awaiting_reply` / `idle` | ничего | пропустить |

## Процедура режима `tick` (одноразовый проход)

1. `gennady inbox --json [--vcs-source=<host>]` — один проход (обновляет реестр).
2. Подача по «Презентация»: есть дельта → виджет; пусто → эмодзи-строка `😴 Без изменений…`.
   БЕЗ диалога, ничего не постишь, Ask не задаёшь. Это ровно то, что повторял бы loop.

## Процедура режима `list`

1. **Показать инбокс.** `gennady inbox --json [--vcs-source=<host>]` → отрисуй
   виджет изменений (см. «Презентация»).

2. **Ask: что берём.** Через `AskUserQuestion` предложи ОДИН пункт из `[ответить]`/
   `[ревью]` (по срочности; `[ответить]` важнее). Беру один → работаем с ним.

3. **Собрать фактуру заново** (out — свежий результат, не кэш):
   1. Подними worktree: `gennady vcs-worktree --ref <group/project!iid> [--vcs-source=<host>]`.
      Команда напечатает `path` (рабочая копия), `base` (для локального дифа) и
      `diff_refs` (`base`/`start`/`head` — нужны для line-комментов).
   2. Возьми пакет: `gennady inbox --pick <ref> [--vcs-source=<host>]` — стадия,
      роль, открытые вопросы.
   3. Прочитай нужный код в `path` (read-only) для честного факт-чека.
   4. **Подтяни существующее обсуждение** — что уже написали другие, где меня
      упомянули, что я сам уже писал:
      - `gennady review-issues --ref <group/project!iid> --all [--vcs-source=<host>]` —
        все треды MR (не только unresolved). Артефакт `<Review_Threads>`: на каждый
        тред `file`/`line`/`resolved` и реплики с ролью `Reviewer` (чужие) / `Author`
        (автор MR) / `AI_Agent` (🤖-заметки бота), у каждой `uid` = логин.
      - `gennady review-issues --ref <...> --draft [--vcs-source=<host>]` — мои
        неопубликованные черновики по этому MR.
      Это вход для директивы (см. шаг 4): чужие доводы могут уже объяснять сущность,
      моё/чужое/ботовское уже сказанное дублировать нельзя, упоминания меня — ждут ответа.

4. **Ввести меня в контекст — визуально, не стеной текста.**

   **Сначала — карта изменений, ВСЕГДА (инвариант 1), независимо от размера MR.**
   Посчитай `git -C <path> diff --numstat <base>..HEAD` + `--name-status` и покажи в чате
   компактную карту: затронутые файлы структурой по папкам/модулям, у каждого тип
   (🆕 новый · ✏️ изменён · 🗑️ удалён · ↪︎ переименован) и объём (± строк), сверху строка
   итога (файлов · +X/−Y · сколько новых файлов), снизу 1–2 фразы сути. Транспорт: ASCII-
   дерево + эмодзи-иконки (надёжно и компактно); виджет — если уместно и доступен. Для
   крошечного MR это короткая карта на несколько строк, но она есть всегда. Я должен
   понять «что и как изменено» из чата, не открывая MR.

   **Затем — для крупных/архитектурных MR — полный разбор.** Если MR вводит
   архитектуру (новые сущности/сервисы/сторы/потоки) или есть design doc — поверх карты
   применяй директиву `ai/directives/agent-inbox/arch-interrogation.directive.xml` ко всему
   изменённому коду: она обстукивает каждую сущность батареей вопросов (зачем, нет ли
   аналога, как принято здесь, YAGNI ли, зачем граница) и выдаёт helicopter-вью.
   Передай в директиву `ref` + `webUrl` (из `inbox --json`/`--pick`), `diff_refs`,
   `path` + `base` (из `vcs-worktree` — по ним директива считает размер изменений:
   `git -C <path> diff --numstat <base>..HEAD` + `--name-status`), а также
   `prior_threads` (артефакт `review-issues --all` из шага 3.4), `my_drafts`
   (`review-issues --draft`) и `my_login` — директива по `AX_PRIOR_DISCUSSION_AWARENESS`
   не дублирует уже сказанное и помечает треды, ждущие моего ответа. Отчёт обязан
   **открываться и закрываться ссылкой на MR** (`ref` → `webUrl`), а сразу под шапкой —
   нести **scope-итог** (`AX_CHANGESET_SCALE`): размер (малый/средний/крупный), сколько
   файлов/строк ±, сколько новых файлов и сущностей, разбивка по категориям
   (код/тесты/доки/конфиг/ассеты) — чтобы было видно объём и характер (расширение vs
   правка), а не только ссылку.
   Подача (см. директиву, OutputFormat): шапка со ссылкой → scope-итог (размер +
   разбивка по категориям) → helicopter-диаграмма (C4) → flow-диаграммы
   нетривиальных сценариев → карточки на каждую сущность с вердиктом
   (`justified`/`weak`/`unjustified`) → связи (визуализировать, не прозой) → итог.
   Что можно показать связью элементов — показывай диаграммой (`mcp__visualize__show_widget`),
   текст оставляй для одиночных суждений. Дальше я задаю доп. вопросы — отвечай по коду/треду.

5. **Подготовить.**
   - `reply_needed`: на каждый открытый вопрос — **честный факт-чек**: в чём
     собеседник прав, в чём нет (по треду + коду). Затем **один краткий ответ** от
     меня (не варианты). Где не уверен — ⚠.
   - `review_needed`: применённая на шаге 4 директива `arch-interrogation` уже дала
     вердикты по сущностям и сверила их с существующими тредами. Кандидаты бывают
     трёх видов (`kind`):
     - **new-line-comment** — новое замечание (`weak`/`unjustified`), привязанное к
       строкам дифа;
     - **reply-to-thread / correction-reply** — это уже поднято другим (или ботом, или
       мной в черновике): не дубль, а reply в существующий тред (`{discussionId}`) —
       либо усилить, либо аргументированно поправить чужой вывод;
     - **awaiting-my-reply** — тред, где меня упомянули или где после моей реплики уже
       ответили: показать отдельно, нужен мой ответ.
     Директива сама НЕ постит — постинг остаётся за скиллом (шаг 6–7).

6. **Согласование (Ask на каждый ответ/замечание).** Покажи предлагаемый ответ →
   `AskUserQuestion`: запостить / поправить / пропустить. Я могу скорректировать текст.

7. **Постинг (стадия B).** После approve — `vcs-reply`, читает JSON-массив из stdin.
   Типы элементов:
   - ответ в тред: `{"discussionId":"<id>","body":"..."}`;
   - новый тред: `{"body":"..."}`;
   - **комментарий на строку** (ревью): `{"body":"...","position":{"baseSha","startSha",
     "headSha","newPath","newLine"}}`. SHA — из `diff_refs` (печатает `vcs-worktree`).
     Правило линий: **added** → `newLine`; **removed** → `oldLine`; **context** → ОБЕ.
     `oldPath` = `newPath`, если файл не переименован.

   **Как запускать** (host берётся из `--vcs-source`; иначе — из origin текущего репо):
   ```
   # 1) собрать комментарии в файл, прогнать dry-run (ничего не постит):
   gennady vcs-reply --project=<group/project> --iid=<iid> --vcs-source=<host> --dry-run < comments.json
   # 2) показать мне результат dry-run; после approve — боевой (без --dry-run):
   gennady vcs-reply --project=<group/project> --iid=<iid> --vcs-source=<host> < comments.json
   ```
   Боевой режим требует `GITLAB_PERSONAL_TOKEN`. Диаграммы в `body` — только Mermaid
   (см. жёсткое правило 5).

8. **Закрыть.** Сними worktree: `gennady vcs-worktree --cleanup <path>`. Запиши итог
   разбора в `~/.gennady/inbox-out/<project>!<iid>.md` (что обсуждали, что запостили).

## Что вне скоупа

- Роль автора своих MR (ответы на ревью моих MR) — позже.
- Запуск тестов/сборки MR (исполнение чужого кода) — нужна docker-изоляция, не сейчас.
- GitHub — только GitLab.
