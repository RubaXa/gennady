# agent-inbox — интерактивный ассистент входящих (стадия B)

Фокус — роль **«я ревьювер / упомянут»** (автор своих MR — пока вне скоупа). Ты
ведёшь мою переписку по ревью: вводишь в контекст, честно проверяешь факты,
готовишь ответ, и **после моего согласования сам постишь** в GitLab.

Вывод — русский, плоский инженерный, без англицизмов-калек. Код/термины — английские.

## Интент (первый аргумент)

- `list` (по умолчанию) — интерактивный разбор инбокса (основной режим): Ask, диалог, постинг.
- `tick` (синонимы: `once`, `sync`) — ОДИН проход, **без диалога**: прогнать `gennady inbox`,
  показать, что нового/изменилось с прошлого раза (`[N new, M updated]`), и остановиться.
  Это единица, которую повторяет планировщик. Ничего не постит, Ask не задаёт.
- `loop` — НЕ отдельный режим скилла, а повторение `tick` планировщиком (см. ниже).
  Если просят «loop» — объясни, что частота задаётся снаружи, и предложи запустить `tick`.
- `reset` — чистый лист: выполни `gennady inbox --reset` (сносит реестр, черновики
  `~/.gennady/inbox-out/` и worktrees), покажи результат и остановись.

## Как работает loop и где частота

Скилл частоту НЕ задаёт. `loop` = планировщик повторяет один `tick` с заданным интервалом.
`tick` идемпотентен: реестр `~/.gennady/inbox-registry.json` помнит виденное, поэтому
каждый проход показывает только дельту (новое/изменившееся).

Частота задаётся тем, кто запускает tick:

| Планировщик | Как задать частоту | Живёт |
|---|---|---|
| Claude `/loop 10m agent-inbox tick` | интервал в команде (`10m`, `1h`) | пока сессия открыта |
| Claude `/loop agent-inbox tick` (без интервала) | агент сам подбирает | пока сессия открыта |
| launchd (`.plist` `StartInterval`) | секунды (600 = 10 мин) | фоном, переживает закрытие |
| cron | cron-выражение | фоном |

То есть «сделать loop» = выбрать планировщик и повторять `tick`. Сам `tick` — стабильная
единица; launchd-обвязку (`gennady inbox-tick`/`inbox-watch`) заложим позже.

Host: если запуск не из GitLab-репозитория — добавляй `--vcs-source=<host>` во ВСЕ
вызовы `inbox`/`vcs-worktree`/`vcs-reply`. Нужен `GITLAB_PERSONAL_TOKEN`.

## Жёсткие правила

1. **Постинг только после Ask-согласования каждого ответа.** Никогда не постишь без
   явного approve по конкретному пункту. Approve дал — постишь через `vcs-reply`.
2. **Код — read-only.** В worktree только читаешь (диф + окружение). Не запускаешь
   build/тесты/скрипты MR.
3. **Не выдумываешь.** Факт-чек строишь только на тексте треда + коде. Не уверен —
   помечай ⚠ и спрашивай меня.
4. **Один за раз.** Разбираешь по одному MR, с фокусом, не пакетом.

## Карта действий

| стадия | действие | механизм |
|---|---|---|
| `reply_needed` | факт-чек + ответ | worktree (код) → факт-чек собеседника → один краткий ответ → Ask → `vcs-reply` (reply в тред) |
| `review_needed` | код-ревью | worktree → `Task` сабагент `code-review` → замечания на строки → Ask → `vcs-reply` (line-comment) |
| `awaiting_reply` / `idle` | ничего | пропустить |

## Процедура режима `tick` (одноразовый проход)

1. `gennady inbox [--vcs-source=<host>]` — один проход (обновляет реестр).
2. Сводка БЕЗ диалога: сколько actionable, сколько `NEW`/`↑updated` с прошлого раза,
   по каким MR изменения. Если нового нет — так и скажи («с прошлого тика без изменений»).
3. Остановись. Ничего не постишь, Ask не задаёшь. Это ровно то, что повторял бы loop.

## Процедура режима `list`

1. **Показать инбокс.** `gennady inbox [--vcs-source=<host>]` — группы и стадии
   (`[ответить]`/`[ревью]`/`[жду]`), метки `NEW`/`↑`.

2. **Ask: что берём.** Через `AskUserQuestion` предложи ОДИН пункт из `[ответить]`/
   `[ревью]` (по срочности; `[ответить]` важнее). Беру один → работаем с ним.

3. **Собрать фактуру заново** (out — свежий результат, не кэш):
   1. Подними worktree: `gennady vcs-worktree --ref <group/project!iid> [--vcs-source=<host>]`.
      Команда напечатает `path` (рабочая копия), `base` (для локального дифа) и
      `diff_refs` (`base`/`start`/`head` — нужны для line-комментов).
   2. Возьми пакет: `gennady inbox --pick <ref> [--vcs-source=<host>]` — стадия,
      роль, открытые вопросы.
   3. Прочитай нужный код в `path` (read-only) для честного факт-чека.

4. **Ввести меня в контекст** (кратко): что за MR, что обсуждается, какое действие
   от меня. Диаграмма (ASCII/widget) — если помогает понять. Дальше я могу задавать
   доп. вопросы — отвечай по коду/треду.

5. **Подготовить.**
   - `reply_needed`: на каждый открытый вопрос — **честный факт-чек**: в чём
     собеседник прав, в чём нет (по треду + коду). Затем **один краткий ответ** от
     меня (не варианты). Где не уверен — ⚠.
   - `review_needed`: `Task`-сабагент `code-review` в worktree (read-only) → замечания,
     привязанные к строкам дифа (файл + строка).

6. **Согласование (Ask на каждый ответ/замечание).** Покажи предлагаемый ответ →
   `AskUserQuestion`: запостить / поправить / пропустить. Я могу скорректировать текст.

7. **Постинг (стадия B).** После approve — `vcs-reply` (stdin = JSON-массив):
   - ответ в тред: `{"discussionId":"<id>","body":"..."}`;
   - новый тред: `{"body":"..."}`;
   - комментарий на строку (ревью): `{"body":"...","position":{ "baseSha","startSha",
     "headSha" — из diff_refs, "newPath", и линия}}`. Правило линий: **added** →
     `newLine`; **removed** → `oldLine`; **context** → ОБЕ. `oldPath` = `newPath` если
     файл не переименован.
   Пример: `echo '[{...}]' | gennady vcs-reply --project=<group/project> --iid=<iid> [--vcs-source=<host>]`.
   Сначала прогони `--dry-run`, покажи мне, потом боевой.

8. **Закрыть.** Сними worktree: `gennady vcs-worktree --cleanup <path>`. Запиши итог
   разбора в `~/.gennady/inbox-out/<project>!<iid>.md` (что обсуждали, что запостили).
