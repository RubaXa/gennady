# agent-inbox

> Статус: research / draft · Дата: 2026-06-24 · Источник: prd-interview

Локальный фоновый агент-диспетчер задач. Задачи **копятся** в файловой
очереди, потребитель их **забирает** (claim) и выполняет. Старт — один тип
задачи: ревью GitLab MR. Архитектура инкрементальная: интерфейс
«продюсер → очередь → потребитель» стабилен и расширяем под новые типы задач.

Это исследовательский документ (PRD), а не финальная спека. Спека скоупа —
позже в `specs/agent-inbox/`.

## 1. Проблема

Разработчик упускает то, что ждёт его внимания в GitLab: где он назначен
ревьювером, где его упомянули, где в его MR появились новые вопросы. Сейчас это
ловится вручную и с задержкой. Нужен локальный фоновый процесс, который
периодически сам находит такие места и готовит разработчику ответ/ревью — без
публичных действий от его имени. Долгосрочно это не только про ревью, поэтому
ядро строим как обобщённый диспетчер задач, а ревью — первый и единственный
реализованный тип задачи.

## 2. Цели и не-цели

**Цели**

- Локальный фоновый процесс, переживающий закрытие приложения, периодически
  проверяет GitLab и находит MR, где пользователь автор / ревьювер / упомянут.
- Найденное превращается в задачи и копится в файловой очереди; продюсер (скан)
  и потребитель (исполнитель) развязаны.
- Потребитель забирает задачу (claim, не просто чтение) и для ревью-задач
  запускает анализ, формируя готовый ответ/ревью **для пользователя**.
- Инкрементальная архитектура: интерфейс «продюсер → очередь → потребитель»
  стабилен и расширяем без переписывания.
- Раздача другим пользователям через шаримый скилл/конфиг.

**Не-цели (явно не делаем в этой итерации)**

- Правила ревью под конкретный проект (разные guidelines).
- Авто-постинг комментариев в GitLab от имени пользователя.
- Другие типы задач («ответь в чате» и пр.) — только шов под них.
- GitHub-путь — только GitLab сейчас.

## 3. Пользователи и сценарии

| Роль                   | Что хочет                                               | Главный сценарий                                                               |
| ---------------------- | ------------------------------------------------------- | ------------------------------------------------------------------------------ |
| Разработчик (владелец) | Не пропускать ревью/упоминания, получать готовый разбор | Фон раз в N минут находит новый MR на ревью → готовит разбор и черновик ответа |
| Другой разработчик     | Поставить себе такой же агент                           | Берёт шаримый скилл/конфиг, минимально настраивает, запускает у себя           |

## 4. Требования (EARS)

| ID     | Тип          | Требование                                                                                                                                                                    | Как проверить                                                             |
| ------ | ------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------- |
| REQ-01 | event-driven | WHEN срабатывает расписание, the scan-команда shall запросить GitLab и вернуть MR, где пользователь автор, ревьювер или упомянут, обновлённые после сохранённого курсора.     | Мок GitLab API, прогнать скан с заданным курсором, проверить выборку.     |
| REQ-02 | event-driven | WHEN найден подходящий MR, the scan-команда shall поставить в очередь задачу с типом `review` и ссылкой на MR.                                                                | После скана прочитать файл очереди, проверить наличие задачи.             |
| REQ-03 | unwanted     | IF задача для пары (MR + курсор) уже есть в очереди или выполнена, THEN the scan-команда shall не создавать дубликат.                                                         | Прогнать скан дважды без новых изменений — вторая задача не появляется.   |
| REQ-04 | ubiquitous   | The queue-CLI shall поддерживать операции enqueue, claim и done над файловой очередью.                                                                                        | Юнит-тесты на каждую операцию.                                            |
| REQ-05 | event-driven | WHEN потребитель забирает задачу, the queue-CLI shall атомарно пометить её `in-progress` с арендой (lease) и не отдавать другому потребителю.                                 | Конкурентный claim двумя процессами — задачу получает один.               |
| REQ-06 | unwanted     | IF задача `in-progress` не завершена за таймаут аренды, THEN the queue-CLI shall вернуть её в очередь как доступную.                                                          | Захватить задачу, не завершить, промотать время — задача снова claimable. |
| REQ-07 | event-driven | WHEN потребитель обрабатывает задачу типа `review`, the система shall запустить ревью-скилл и сформировать разбор и черновик ответа для пользователя.                         | Прогон на тестовом MR — на выходе текст разбора.                          |
| REQ-08 | ubiquitous   | The система shall НЕ публиковать комментарии в GitLab — результат отдаётся только пользователю.                                                                               | Проверить отсутствие write-вызовов GitLab API в потребителе.              |
| REQ-09 | event-driven | WHEN задача обработана, the потребитель shall пометить её `done` и обновить курсор скана.                                                                                     | После прогона проверить статус задачи и значение курсора.                 |
| REQ-10 | ubiquitous   | The система shall читать GitLab-токен из переменной окружения `GITLAB_PERSONAL_TOKEN`.                                                                                        | Запуск без переменной — внятная ошибка; с переменной — работает.          |
| REQ-11 | optional     | WHERE задача неизвестного типа, the потребитель shall пропустить её и залогировать, не падая.                                                                                 | Положить задачу с чужим типом — потребитель не падает.                    |
| REQ-12 | ubiquitous   | The система shall выполнять один проход (продюсер + забор и выполнение задачи) единой командой `gennady inbox-tick`, независимой от способа запуска.                          | Вызвать `tick` напрямую из шелла — проход проходит без `/loop`.           |
| REQ-13 | event-driven | WHEN задача `review` выполнена, the потребитель shall записать полный разбор в `.gennady/out/<id>.md` и вызвать `notify(signal)` с короткой сводкой (title, 2–3 строки, ref). | После прогона есть md-файл и вызов notify со сводкой.                     |
| REQ-15 | ubiquitous   | The output sink shall быть единым интерфейсом `notify(signal)`, скрывающим конкретный канал от потребителя.                                                                   | Подменить канал на тестовый — потребитель не меняется.                    |
| REQ-14 | unwanted     | IF `done`-запись старше 14 дней, THEN the система shall удалить её из архива.                                                                                                 | Подсунуть старую запись, прогнать чистку — запись исчезает.               |

Типы EARS: `ubiquitous` · `event-driven` (WHEN) · `state-driven` (WHILE) ·
`optional` (WHERE) · `unwanted` (IF/THEN).

## 5. Технический подход

Центральная абстракция — **файловая очередь задач**, продюсер и потребитель
развязаны (это и есть инкрементальный шов).

```
┌── ПРОДЮСЕР ──────────┐   ┌── ОЧЕРЕДЬ ──────┐   ┌── ПОТРЕБИТЕЛЬ ────────┐
│ scan GitLab          │──▶│  tasks-файл     │──▶│ skill: claim задачу  │
│ автор/ревьювер/      │   │  CLI: enqueue / │   │ → review-скилл       │
│ упомянут, since=курс. │   │  claim / done   │   │ → разбор+ответ мне   │
└──────────────────────┘   └─────────────────┘   └──────────────────────┘
        ▲                          ▲                        ▲
        └── узкий старт (review) ──┴── стабильный API ───────┘
            расширяемо: др. продюсеры/потребители — потом
```

Переиспользуем из `gennady`:

- `review-issues --since <ISO>` и `computeCursor` — выборка «что нового» готова.
- VCS-клиент GitLab (`services/vcs-client`: MergeDiscussions, пагинация).
- Паттерн polling из `services/agent-mon` (`observe({interval})`) и
  оркестраторные паттерны ai-skills.

**Единица работы — runtime-независимая CLI-команда `gennady inbox-tick`**: один
проход = «прогнать продюсера + забрать и выполнить задачу». И `/loop`, и launchd
зовут ровно её, поэтому миграция с прототипа на демон сводится к смене триггера,
а не переписыванию.

Запуск (для исследования): **`/loop` в открытой сессии** дёргает `tick`, пока
сессия открыта. Это осознанно прототипный runtime; для 24/7 позже триггер
меняется на launchd, та же `tick` — кода не трогаем.

**Вывод разбора** идёт через подключаемый канал (output sink) — ещё один шов.
Интерфейс — единственный метод `notify(signal)`, где `signal = {title, summary,
ref}`: полный разбор лежит в файле, наружу уходит **короткая сводка** (ping + 2–3
строки сути + ссылка). Потребитель не знает, какой канал активен.

- Сейчас канал один: чат сессии.
- Первый канал после чата — **нотификация macOS** (osascript/terminal-notifier):
  локально, без токенов и сети, поэтому короткая сводка безопасна. Сама интеграция
  — out of scope этой итерации, закладываем интерфейс.
- Мессенджер (Telegram/Slack) — позже; там короткая сводка уходит наружу, нужно
  отдельно решать про утечку.

## 5a. Реализовано (спайк)

Первый кирпич продюсера готов и проверен на живых данных (`gitlab.corp.mail.ru`,
144 todo/MR на входе).

**Клиент — `client.Inbox.getActionable()`** (GitLab, GraphQL):

- один запрос `currentUser { todos(type: MERGEREQUEST) + reviewRequestedMergeRequests
  - authoredMergeRequests }` — закрывает обе стороны: «от меня ждут» и «я жду»;
- GraphQL-транспорт отдельный (`POST /api/graphql`), т.к. REST-клиент прибит к `/api/v4`;
- чистая нормализация в `VcsActionableMr`: одна `role` (приоритет
  `author > reviewer > mentioned`), события-состояния в `events[]`, флаг
  `directlyAddressed`; дедуп по `webUrl`. Никакой политики отсева в клиенте.

**Команда — `gennady inbox`** (host автодетектится из origin):
вся политика здесь, клиент остаётся сырым источником.

- **Группировка по роли**, не по проекту: `Ждут моё ревью` / `Мои MR` /
  `Меня упомянули`. Проект — метка в строке.
- **События — флаги, не строки.** CI/mergeable (`build_failed→ci_failed`,
  `unmergeable`) показываются **только для роли author**. «Без разницы, что упал CI
  в MR, где я ревьювер» — закрыто.
- **Отсев по умолчанию** (всё переключаемо): нет роли → не показываем (жёстко,
  5 шт. на входе); черновики скрыты (`--drafts`); элементы без апдейта >14 дней →
  stale, скрыты (`--include-stale` / `--stale-days=N`); исключение для черновиков
  — только свежий author-черновик с CI/unmergeable.
- Флаги: `--drafts`, `--include-stale`, `--stale-days=N`, `--ci-all`, `--all`.
- Шапка честно показывает, сколько и почему скрыто.

**Результат на живых данных:** 144 → **20** видимых (17 ревью + 3 мои MR), скрыто
84 stale + 35 drafts + 5 noise.

**Файлы:** клиент — `services/vcs-client/{abstract/vcs-client-inbox.ts,
gitlab/vcs-gitlab-inbox.ts, entities/vcs-actionable-mr.type.ts}`; команда —
`cli/cmd/inbox/` (`inbox.cmd.ts` + `_core/logic/{build-inbox-context,
build-inbox-view,render-inbox-view}`).

**Карта действий и исполнение (кирпич 6):** действия описаны явной картой в
`ai/skills/agent-inbox/SKILL.md` — оркестратор не делает работу сам, а диспатчит
`Task`-сабагента (как `sdd-execute`):

```
review_needed → worktree клона (repos.json) → Task: code-review в worktree → черновик
reply_needed  → Task: ответы на open questions из --pick → черновик
awaiting/idle → ничего
```

Код-ревью требует реального кода. Команда **`gennady vcs-worktree`** (готова,
git-механика проверена на репозитории gennady): находит клон (repos.json → скан
`~/Developer` по origin → shallow-клон в `~/.gennady/clones/`), берёт head MR
(`git fetch origin merge-requests/<iid>/head`), создаёт detached `git worktree` с
**выключенными хуками** (read-only, ничего не исполняется), печатает путь + `git
diff`-строку; `--cleanup <path>` сносит. Сабагент `code-review` работает в этом
worktree только на чтение. Полная цепочка на реальном MR ещё не прогнана (нужен
доступ к клонам под `~/Developer`).

**Реестр и дельта (кирпич 2, готово):** `inbox` помнит виденное в глобальном
реестре `~/.gennady/inbox-registry.json` и помечает каждый MR `NEW` (впервые) или
`↑ updated` (`updatedAt` сдвинулся с прошлого тика); шапка показывает `[N new,
M updated]`. Флаг `--no-save` — read-only прогон. На живых данных: тик 1 → 19 new,
тик 2 подряд → 0 new (всё idle). Это и есть «получать только то, что изменилось».
Дельта пока грубая (по `updatedAt`); точный сигнал «мне ответили в треде» — кирпич 3.

**Чего ещё нет** (следующие кирпичи): точная классификация стадий через скан
обсуждений (кирпич 3), file-очередь и `tick`, запуск ревью, output sink.

## 5b. Машина состояний MR и стадии автономии

Конечная цель — ассистент, который ведёт переписку по MR. Каждый MR проходит
стадии; на каждом тике стадия пересчитывается из инбокса + реестра + (позже) скана
обсуждений:

```
NEW (role=reviewer, нет моего ревью)   → REVIEW_NEEDED → REVIEW_PREPARED (notify)
обновление, мне ответили/упомянули     → REPLY_NEEDED  → варианты ответа (notify)
ушёл из инбокса / нет нового            → IDLE
```

**Автономия — поэтапно, граница явная:**

- **A (текущая):** агент только готовит черновик/ответ и уведомляет; в GitLab НЕ
  пишет. Постит человек.
- **B:** постит после явного OK на конкретный item.
- **C:** автономно там, где заранее делегировано (проект/лейбл).

Сейчас реализуется только стадия A. B/C — отдельные осознанные шаги, не по умолчанию.

**Модель исполнения — гибрид:** CLI собирает контекст, разбор/ответ генерит агент
в сессии. CLU на TS сам LLM-разбор не делает.

- CLI: `gennady inbox --pick <group/project!iid>` отдаёт **рабочий пакет** одного MR
  — стадия, роль и «открытые вопросы» (ноты других после моего последнего, т.е. что
  именно надо ответить); для `review_needed` — флаг «первый ревью».
- Агент в сессии (тот, что крутит `/loop`): берёт пакет, делает черновик ответа/ревью,
  пишет в `.gennady/out/`, уведомляет (стадия A — без записи в GitLab).

Кирпич 6a (сборка пакета) — готово, проверено на живых данных (`!510`: reply_needed,
5 открытых вопросов). Кирпич 6b (черновик в сессии + notify) — следующий.

**Стадии через скан обсуждений (кирпич 3, готово):** для каждого видимого MR с
дельтой `new`/`updated` команда тянет обсуждения (`MergeDiscussions.getAll`) и
сравнивает мою активность с чужой (identity — `client.getCurrentUser()`, REST
`GET /user`):

- `review_needed` — я reviewer и ещё ничего не писал;
- `reply_needed` — кто-то написал после меня (или я ни разу не ответил) → мяч у меня;
- `awaiting_reply` — я ответил последним, жду;
- `idle` — реагировать не на что.

Idle-MR не сканируются повторно — стадия берётся из реестра. В выводе стадия —
тег (`[ответить]`/`[ревью]`/`[жду]`), сортировка внутри группы — по срочности
(`reply_needed` наверх). На живых данных: identity `k.lebedev`, в группе ревью
видно `[ответить]` (мне ответили), `[ревью]` (не смотрел), `[жду]` (мой мяч сыгран).

## 5c. Команды (нейминг)

Корень семьи — `inbox` (совпадает с именем сервиса, универсален: «входящие»
любого источника, не только MR). Структура — плоский префикс, как `review` /
`review-issues` / `review-verify`.

| Команда                 | Что делает                                                     | Статус       |
| ----------------------- | -------------------------------------------------------------- | ------------ |
| `gennady inbox`         | показать, что требует моей реакции (группы/стадии/дельта)      | готово       |
| `gennady inbox-tick`    | один проход оркестратора (scan → очередь → обработка → notify) | план (к.7)   |
| `gennady inbox-queue`   | очередь задач: положить/забрать/завершить                      | план (к.4)   |
| `gennady inbox-watch`   | крутить проход по интервалу (обёртка `/loop`)                  | план (к.8)   |
| `gennady vcs-worktree`  | git worktree клона на ветке MR для ревью (prepare/cleanup)     | план (к.6)   |
| `gennady inbox-install` | поставить фоновый запуск (launchd)                             | план (later) |

`mr-inbox` переименован в `inbox` — старое имя прибивало универсальный движок к MR.

## 5d. Рабочий процесс v2 (интерактивный, согласован)

Фокус — роль **«я ревьювер/упомянут»** (автор — позже). Стадия **B**: агент постит
сам, но только после Ask-согласования каждого ответа.

```
inbox list                      → список actionable (review_needed / reply_needed)
  └─ Ask: «с чем работаем?»      → выбираю ОДИН пункт
       └─ агент собирает фактуру ЗАНОВО (out = свежий результат, не кэш):
            • поднимает worktree клона (для факт-чека по коду + диф)
            • краткое введение: что за MR, что обсуждается, действие от меня;
              диаграмма — если помогает понять
            • reply_needed: честный факт-чек ответа собеседника (в чём прав / нет)
              по тексту треда + коду; + ОДИН краткий предлагаемый ответ
            • review_needed: код-ревью → замечания, привязанные к строкам дифа
       └─ агент вводит в контекст → я задаю доп. вопросы (диалог)
       └─ на каждый ответ/замечание → Ask на согласование
            └─ approve → агент постит через vcs-reply
                 (новый коммент в тред ИЛИ дискуссия на строку дифа)
       └─ worktree чистится; out остаётся артефактом в ~/.gennady/inbox-out/
inbox loop                      → тот же tick в фоне (позже)
```

Что прокачать под это: `vcs-reply` (создавать новую дискуссию + line-comment на
диф), скилл `agent-inbox` (переписать процедуру под интерактив + per-answer approve

- постинг), стадия B как явный гейт. Текущий скилл (пакетный, стадия A) — это v1.

## 6. Данные и контракты

- **Очередь задач** — активный файл (например `.gennady/tasks.jsonl`) держит только
  `pending`/`in-progress`. Запись:
  `{id, type, ref (MR), status: pending|in-progress|done, lease_until, created_at}`.
  Владелец — queue-CLI.
- **Архив** — `done`-задачи переносятся в архивный файл (`.gennady/tasks.archive.jsonl`);
  записи старше **14 дней** чистятся. Активный файл не растёт от выполненных задач,
  dedup и чтение не деградируют.
- **Курсор скана** — ISO-timestamp последнего обработанного обновления
  (`computeCursor`), хранится в состоянии (например `.gennady/scan-cursor.json`).
- **Вывод** — markdown-разбор на задачу пишется в `.gennady/out/<id>.md` (переживает
  сессию). Наружу через `notify(signal)`: `signal = {title, summary (2–3 строки),
ref}`; канал подключаемый, по умолчанию — сводка в чат.
- **Actionable inbox** — `VcsActionableMr {iid, project, webUrl, title, updatedAt,
draft, role, events[], directlyAddressed}`. Сырой нормализованный факт; `stale` и
  отсев — производные в слое команды, не в сущности.
- **Реестр инбокса** — глобальный `~/.gennady/inbox-registry.json`, ключ — webUrl:
  `{project, iid, role, stage, lastSeenUpdatedAt, firstSeenAt, lastClassifiedAt}`.
  Инбокс охватывает десятки проектов, поэтому реестр глобальный, не пер-репо. Хранит
  то, чего нет в GitLab: что агент уже видел/делал. Дельта — по `updatedAt`.
- **Внешний контракт** — GitLab read-only: GraphQL (`/api/graphql` — actionable
  inbox) + REST (`/api/v4` — MR, discussions). Запись в GitLab не используется.

## 8. Риски и безопасность

| Риск                                 | Вероятность/урон                   | Митигация                                                                                                             |
| ------------------------------------ | ---------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| ⚠ Токен GitLab локально              | средн./высок.                      | ENV `GITLAB_PERSONAL_TOKEN`, не коммитить, не логировать значение; для раздачи — инструкция, не файл с токеном в репо |
| ⚠ Гонки/дубли в файловой очереди     | средн./средн.                      | claim с lease + dedup по (MR+курсор) (REQ-03/05/06)                                                                   |
| `/loop` умирает с сессией → пропуски | высок./низк. (для исследования ок) | принято для прототипа; план миграции на launchd                                                                       |
| Лишние/пустые прогоны при polling    | средн./низк.                       | курсор отсекает «ничего нового», скан дешёвый                                                                         |
| Раздача утянет чужой токен/конфиг    | низк./высок.                       | в шаре только скилл+шаблон, секреты — у каждого свои в ENV                                                            |

Security-решения: токен — **принят риск явно** через ENV (стандартно); публичные
действия (авто-постинг) — **закрыты** (не делаем); PII/обхода authz нет.

## 9. Метрики успеха и провала

На этой итерации метрики **не вводим** — не на чем калибровать пороги. Гоняем на
живых MR, оцениваем качественно. Ориентиры на будущее (не контрактные сейчас):

- **Успех:** для нового MR, где пользователь ревьювер/упомянут, агент в течение
  одного тика готовит полезный разбор без ручного запуска; дублей нет.
- **Провал:** пропускает MR, которые должен поймать, плодит дубли/мусор, или
  разбор настолько неточен, что быстрее сделать вручную.

## 11. План реализации

```
[queue-CLI: enqueue/claim/done + lease + архив/чистка]
        │
        ├─→ [scan GitLab → enqueue review-задач + курсор + dedup]
        │           │
        │           └─→ [потребитель: claim → review-скилл → разбор → out + done]
        │
        ├─→ [gennady inbox-tick] ── зовут одинаково ──┐
        │                                             │
        └─→ [/loop сейчас] ───── миграция ──────▶ [launchd потом]
                    │
                    └─→ [шаримый скилл/шаблон конфига]
```

1. ✅ **actionable inbox** — `client.Inbox.getActionable()` (GraphQL) + `gennady
inbox` (группировка/отсев). Готово, проверено на живых данных.
2. ✅ **реестр + дельта** — глобальный реестр виденного, метки `NEW`/`↑ updated`
   по `updatedAt`. Готово.
3. ✅ **стадии через скан обсуждений** — `review_needed`/`reply_needed`/`awaiting_reply`
   на базе `MergeDiscussions.getAll` + identity (`GET /user`); скан только для
   изменившихся видимых MR, idle берёт стадию из реестра. Готово.
4. **queue-CLI** — файловая очередь с `enqueue/claim/done`, lease-таймаут,
   атомарность, архив + чистка >14 дней (REQ-04/05/06/14).
5. **scan** — на базе стадий: enqueue review-задач по курсору с dedup (REQ-01/02/03).
   6a. ✅ **рабочий пакет** — `inbox --pick <ref>`: стадия + открытые вопросы по MR.
   6b. 🟡 **потребитель (в сессии)** — скилл `agent-inbox` (`ai/skills/agent-inbox/`)
   готов как процедура стадии A: `inbox` → топ reply/review → `--pick` → черновик в
   `.gennady/out/` → сводка в чат, без записи в GitLab. Осталось обкатать на живых MR.
6. **`gennady inbox-tick`** — единый проход, runtime-независимый (REQ-12).
7. **триггер** — `/loop` дёргает `tick` (прототип); позже — launchd на ту же `tick`.
8. **раздача** — шаримый скилл + шаблон конфига (ENV, пути), инструкция запуска.
