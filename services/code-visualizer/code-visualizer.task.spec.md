# Техническое задание: Универсальный сервис визуализации кода (`code-visualizer`)

## 1. Назначение системы

Сервис `code-visualizer` предназначен для генерации читаемых визуальных артефактов из:

- исходного кода;
- фрагмента файла из реального проекта;
- unified git diff;
- конфликтов слияния;
- в перспективе других производных представлений кода.

Ключевая цель сервиса: превращать код и diff в изображения, которые удобно читать в мессенджерах, прежде всего в Telegram, а также использовать в review-сценариях, CLI и других автоматизациях.

Сервис не должен проектироваться как Telegram-специфичный helper и не должен быть привязан только к code review. Review — это лишь один из сценариев использования.

---

## 2. Проблема, которую решает сервис

Текстовый код в мессенджерах читается плохо:

- ломается форматирование;
- длинные фрагменты превращаются в нечитаемую простыню;
- unified diff почти невозможно комфортно просматривать на мобильных устройствах;
- line numbers, file metadata и границы изменений теряются.

Поэтому нужен отдельный доменный сервис, который умеет:

1. Нормализовать входной кодовый материал.
2. Построить визуальную модель документа.
3. Разбить длинный контент на страницы.
4. Сгенерировать готовые изображения и сопутствующие метаданные.

---

## 3. Архитектурная позиция

Сервис должен проектироваться как **универсальный визуальный движок**, а не как сквозная обвязка вокруг одного источника данных.

### 3.1. Что входит в сервис

- каноническая внутренняя модель визуального документа;
- преобразование входных структур в эту модель;
- расчёт layout и пагинации;
- рендер итоговых изображений;
- формирование результата, пригодного для отправки в канал доставки.

### 3.2. Что не должно быть жёстко встроено в сервис

- чтение git-репозитория как единственный способ получения данных;
- Telegram API;
- логика review-команд;
- логика конкретного VCS-провайдера.

Эти части должны жить в адаптерах или orchestration-слоях выше сервиса.

---

## 4. Границы ответственности

### 4.1. Что входит в первую целевую область

Сервис обязан поддерживать:

- рендер обычного кода;
- рендер фрагмента файла по диапазону строк;
- рендер unified diff;
- рендер конфликтов слияния;
- line numbers;
- заголовок документа с метаданными;
- пагинацию длинного контента;
- несколько тем оформления;
- возврат как одного изображения, так и набора страниц.

### 4.2. Что не является обязательным для первой версии

Следующие возможности не запрещены, но не должны считаться обязательными в первом рабочем срезе:

- интерактивный HTML viewer;
- split diff view;
- word diff внутри строки;
- inline comments как отдельный визуальный слой;
- OCR/обратный разбор изображений;
- генерация видео или анимаций.

## 6. Основные сценарии использования

### 6.1. Рендер фрагмента исходного кода

Пользователь передаёт:

- строку с кодом или
- путь к файлу и диапазон строк.

Сервис возвращает:

- одно или несколько изображений;
- метаданные страниц;
- сводную информацию о том, что было отрисовано.

### 6.2. Рендер unified diff

Пользователь передаёт:

- raw unified diff text или
- уже нормализованную diff-структуру.

Сервис возвращает:

- визуализацию изменений с различимыми added/deleted/context строками;
- line numbers старой и новой версии;
- разбивку на страницы;
- метаданные для последующей отправки.

### 6.3. Рендер полного файла

Пользователь передаёт:

- путь к файлу или содержимое файла целиком.

Если файл слишком длинный, сервис обязан разбить результат на несколько изображений без потери читаемости.

### 6.4. Визуализация конфликта слияния

Пользователь передаёт:

- текст файла с conflict markers;
- или уже нормализованную структуру конфликта;
- или конфликтующий фрагмент, выделенный из merge/rebase/cherry-pick сценария.

Сервис возвращает:

- визуальное представление конфликтного блока;
- явное разделение секций текущей и входящей версии;
- базовую секцию, если она доступна;
- метки конфликта и контекст соседних строк;
- разбиение на страницы, если конфликт слишком длинный.

## 7. Каноническая модель данных

Сервис не должен напрямую рендерить сырые строки. Сначала вход должен быть преобразован во внутреннюю модель документа.

### 7.1. VisualDocument

Корневой объект визуального документа.

Минимальные поля:

```ts
export type VisualDocumentKind = 'source' | 'diff' | 'conflict';

export type VisualDocument = {
  kind: VisualDocumentKind;
  title?: string;
  subtitle?: string;
  sourceRef?: string;
  theme: VisualThemeId;
  blocks: VisualBlock[];
  meta: VisualDocumentMeta;
};
```

### 7.2. VisualDocumentMeta

```ts
export type VisualDocumentMeta = {
  language?: string;
  filePath?: string;
  revision?: string;
  channelPreset?: string;
  lineCount: number;
  pageCount?: number;
};
```

### 7.3. VisualBlock

Документ должен состоять из блоков.

Минимальный набор типов:

```ts
export type VisualBlock =
  | VisualCodeBlock
  | VisualDiffBlock
  | VisualConflictBlock
  | VisualHeaderBlock
  | VisualParagraphBlock;
```

### 7.4. VisualCodeBlock

```ts
export type VisualCodeBlock = {
  type: 'code';
  lines: VisualCodeLine[];
};
```

### 7.5. VisualDiffBlock

```ts
export type VisualDiffBlock = {
  type: 'diff';
  filePath?: string;
  hunks: VisualDiffHunk[];
};
```

### 7.6. VisualConflictBlock

```ts
export type VisualConflictBlock = {
  type: 'conflict';
  filePath?: string;
  sections: VisualConflictSection[];
};
```

### 7.7. Линия кода

```ts
export type VisualCodeLine = {
  text: string;
  lineNumber?: number;
  tokens?: VisualToken[];
  emphasis?: 'normal' | 'muted' | 'focus';
};
```

### 7.8. Строка diff

```ts
export type VisualDiffLineKind = 'context' | 'added' | 'deleted' | 'hunk' | 'meta' | 'no-newline';

export type VisualDiffLine = {
  kind: VisualDiffLineKind;
  text: string;
  oldLineNumber?: number;
  newLineNumber?: number;
  tokens?: VisualToken[];
};
```

### 7.9. Секция конфликта

```ts
export type VisualConflictSectionKind = 'current' | 'incoming' | 'base' | 'separator' | 'context';

export type VisualConflictSection = {
  kind: VisualConflictSectionKind;
  label?: string;
  lines: VisualCodeLine[];
};
```

### 7.10. Токены

```ts
export type VisualToken = {
  text: string;
  color?: string;
  fontStyle?: 'normal' | 'italic';
  fontWeight?: number;
  backgroundColor?: string;
};
```

## 8. Входные контракты

Публичный API должен принимать разные виды входов, но сводить их к единому внутреннему контракту.

### 8.1. Source input

```ts
export type RenderSourceInput = {
  kind: 'source';
  code: string;
  language?: string;
  title?: string;
  filePath?: string;
  startLine?: number;
};
```

### 8.2. File fragment input

```ts
export type RenderFileFragmentInput = {
  kind: 'file-fragment';
  filePath: string;
  fromLine?: number;
  toLine?: number;
  title?: string;
  language?: string;
};
```

### 8.3. Diff input

```ts
export type RenderDiffInput = {
  kind: 'diff';
  diffText?: string;
  diff?: unknown;
  title?: string;
  filePath?: string;
};
```

### 8.4. Conflict input

```ts
export type RenderConflictInput = {
  kind: 'conflict';
  conflictText?: string;
  conflict?: unknown;
  title?: string;
  filePath?: string;
};
```

### 8.5. Позиция по diff-парсеру

Сервис `code-visualizer` не должен вшивать в себя единственную реализацию diff-парсера.

Он должен уметь работать как минимум в двух режимах:

1. Принимать уже нормализованный diff-документ.
2. Принимать raw unified diff text через адаптер/нормализатор.

Отдельная реализация `services/git-diff-parser` рассматривается как upstream dependency, но не как обязательная часть этого сервиса.

## 9. Выходной контракт

Результат сервиса должен быть пригоден для многостраничной доставки.

```ts
export type RenderedVisualArtifact = {
  kind: 'image-set';
  pages: RenderedVisualPage[];
  summary: RenderedVisualSummary;
};

export type RenderedVisualPage = {
  index: number;
  width: number;
  height: number;
  mimeType: string;
  buffer: Uint8Array | ArrayBuffer;
  caption?: string;
};

export type RenderedVisualSummary = {
  title?: string;
  pageCount: number;
  lineCount: number;
  truncated: boolean;
  warnings: string[];
};
```

Сервис не должен возвращать только “голый buffer”, если контент может занимать несколько страниц.

## 10. Темы, стиль и визуальные пресеты

Сервис должен разделять:

- доменную модель документа;
- базовую тему оформления;
- канал-ориентированные пресеты.

### 10.1. Минимальные темы

Первая версия должна поддерживать:

- `light`
- `dark`

### 10.2. Минимальные пресеты

Первая версия должна поддерживать:

- `default`
- `telegram`

### 10.3. Пресет обязан задавать

- целевую ширину;
- базовый font size;
- line height;
- padding;
- header layout;
- допустимую высоту страницы;
- правила пагинации;
- формат изображения по умолчанию.

---

## 11. Pagination как core feature

Пагинация является обязательной частью системы, а не дополнительной возможностью.

### 11.1. Причины

- длинный файл не помещается в один читаемый кадр;
- Telegram и мобильные устройства требуют крупных шрифтов;
- попытка “впихнуть всё в одну картинку” разрушает полезность сервиса.

### 11.2. Требования

Пагинатор обязан:

- разбивать документ по строкам и блокам;
- не разрывать строку на две страницы;
- по возможности не разрывать hunk в неудачном месте;
- уметь повторять заголовок документа на каждой странице при необходимости;
- выставлять `pageCount` в результате;
- уметь помечать, что результат был усечён.

---

## 12. Требования к diff-визуализации

Для unified diff сервис обязан поддерживать:

- отдельный стиль для `added`, `deleted`, `context`, `hunk`;
- отображение line numbers old/new;
- явное визуальное выделение границ hunk;
- сохранение читаемости даже без идеальной синтаксической подсветки;
- возможность отрисовки как одного файла diff, так и нескольких подряд.

Открытый вопрос для архитектурной сессии:

- должен ли сервис оперировать одним общим `VisualDocument`, где diff — это частный вид блоков;
- или source и diff должны быть соседними моделями с общими примитивами компоновки и рендера.

---

## 13. Требования к визуализации конфликтов

Для конфликтов слияния сервис обязан поддерживать:

- распознавание стандартных git conflict markers:
  - `<<<<<<<`
  - `=======`
  - `>>>>>>>`
- опциональную поддержку блока `|||||||`, если доступна базовая версия;
- отдельный визуальный стиль для каждой стороны конфликта;
- явные подписи сторон конфликта;
- сохранение окружающего контекста строк;
- возможность отрисовать конфликт как специальный документ, а не как обычный diff.

Конфликт нельзя сводить к обычному unified diff без явного решения, так как у него иная семантика:

- есть конкурирующие версии одного участка;
- иногда присутствует базовая версия;
- пользователю важно видеть границы конфликта и принадлежность строк конкретной стороне.

---

## 14. Требования к рендерингу исходного кода

Сервис обязан поддерживать:

- syntax highlighting;
- line numbers;
- header с названием файла или title;
- выделение диапазона строк;
- рендер полного файла;
- fallback, если язык не распознан.

Желательно поддержать:

- акцентные подсветки отдельных строк;
- пометки “focus range”;
- дополнительные подписи и метки.

---

## 15. Нефункциональные требования

### 14.1. Читаемость

Это главный критерий качества. Сервис обязан оптимизироваться под восприятие результата человеком, а не под минимальность выходного файла.

### 14.2. Расширяемость

Нельзя проектировать сервис так, чтобы добавление нового входного типа ломало всю систему.

### 14.3. Производительность

Сервис должен переиспользовать тяжёлые компоненты там, где это уместно:

- экземпляры highlighter;
- fonts;
- themes;
- ресурсы renderer.

### 14.4. Отказоустойчивость

Если вход частично повреждён, сервис должен:

- вернуть warnings;
- по возможности отрисовать частично валидный результат;
- не падать без крайней необходимости.

### 14.5. Детерминизм

При одинаковом входе и одинаковом preset/theme сервис должен выдавать эквивалентный визуальный результат.

---

## 16. Ограничения реализации

1. Сервис не должен быть намертво завязан на Telegram SDK.
2. Сервис не должен зависеть от конкретного CLI entrypoint.
3. Diff parsing и visual rendering должны быть разделены как минимум на уровне модулей.
4. Публичный API не должен раскрывать внутренние layout-детали, которые могут меняться.
5. Контракты должны позволять будущие backend'ы вывода, а не только один renderer.

---

## 17. Точки интеграции в текущем проекте

Первыми потребителями потенциально могут быть:

- review-команды CLI;
- отдельная CLI-команда генерации визуального представления кода, diff или конфликта;
- будущие Telegram-интеграции;
- VCS-flow, где нужно показать кодовый фрагмент или patch в читаемом виде.

Но сам сервис не должен зависеть от этих потребителей.

---

## 18. Открытые архитектурные вопросы

Эти вопросы должны быть проработаны отдельно, до реализации:

1. Как назвать канонический доменный объект: `VisualDocument`, `CodeVisualDocument`, `ArtifactDocument`?
2. Нужна ли одна общая модель для source и diff, или две соседние модели?
3. Где должна жить логика пагинации:
   - в layout engine;
   - в channel preset;
   - в отдельном paginator service?
4. Должен ли сервис принимать raw file paths напрямую или только контент/нормализованные структуры?
5. Является ли diff-парсер обязательной зависимостью или адаптером второго уровня?
6. Нужно ли уже в первой версии предусмотреть annotations/comments layer?
7. Должен ли выход содержать caption templates для каналов доставки?
8. Является ли конфликт частным случаем diff-модели или отдельным видом документа?

## 19. Рекомендуемый target architecture

На текущем этапе рекомендуется проектировать сервис из следующих слоёв:

1. `adapters`
Принимают разные виды входа и приводят их к нормализуемому формату.

2. `normalize`
Строят каноническую внутреннюю модель визуального документа.

3. `highlight`
Добавляют токены, стили и цветовую информацию.

4. `layout`
Рассчитывают размеры, блоки и пагинацию.

5. `render`
Строят сцену и возвращают итоговые изображения.

6. `packaging`
Формируют результат, удобный для следующего слоя доставки.

---

## 20. Первый рабочий срез

Первый рабочий срез должен быть производным от целевой архитектуры, а не отдельным временным решением.

Минимально полезная версия должна уметь:

1. Рендерить source code из строки.
2. Рендерить unified diff из текста.
3. Рендерить конфликт слияния из текста.
4. Показывать line numbers.
5. Поддерживать `dark` и `light`.
6. Поддерживать `telegram` preset.
7. Возвращать набор страниц, а не только одну картинку.

---

## 21. Критерии готовности архитектурной спецификации

Архитектурная спецификация по этому сервису считается достаточной для передачи в реализацию, если в ней зафиксированы:

- ограниченные контексты;
- каноническая модель данных;
- входные и выходные контракты;
- стратегия пагинации;
- стратегия diff/source/conflict rendering;
- точки интеграции с остальными сервисами;
- decision log по спорным архитектурным вопросам;
- первый рабочий срез как подмножество целевой архитектуры.
