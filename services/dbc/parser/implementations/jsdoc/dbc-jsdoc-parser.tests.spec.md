# План тестирования `DbcJsDocParser`

## Цель

Проверить, что `DbcJsDocParser` возвращает `DbcSchema`, строго соответствующую [task-спецификации](services/dbc/parser/dbc-parser.task.spec.md), а snapshot фиксирует именно ожидаемый эталонный результат, а не случайный текущий вывод реализации.

## Базовый подход

ВАЖНО: Строго следуй правилам написания тестов, указанным в `ai/agents/agent-qa-code.rules.xml`

### Принцип тестирования: snapshot как эталон полной схемы

Каждый тест:

1. задает входной контракт прямо в коде теста;
2. вызывает `parse(inputContract)`;
3. сравнивает результат через snapshot целиком как `DbcSchema`.

Snapshot в этом наборе тестов нужен не ради сокращения кода, а ради полного diff-контекста. При падении должно быть сразу видно:

- какие `entries` уже корректны;
- какое поле отличается;
- у какой именно записи `issues` появились, пропали или стоят не на той строке.

### Правила для QA-агента

- snapshot должен соответствовать [спецификации парсера](services/dbc/parser/dbc-parser.task.spec.md), а не просто текущему выводу реализации;
- перед обновлением snapshot нужно вручную сверить ожидаемое поведение со спецификацией;
- snapshot фиксирует уже проверенное ожидание и не заменяет анализ;
- snapshot-файлы должны храниться только в отдельной папке `services/dbc/parser/implementations/jsdoc/__tests__/snapshots`;
- `snapshot.setResolveSnapshotPath(...)` должен настраиваться в едином setup-файле в корне проекта (например, `snapshot-path.setup.ts`) и импортироваться в тестах, а не дублироваться локально в каждой тестовой директории;
- входные данные пишутся прямо в тесте, без входных fixture-файлов;
- тестовые файлы должны быть разложены по группам, чтобы не собирать один монолитный файл на 1000+ строк.

## Структура тестовых файлов

- Рабочая директория `services/dbc/parser/implementations/jsdoc/__tests__`
- Снимки (snapshot) должны лежать только в `services/dbc/parser/implementations/jsdoc/__tests__/snapshots`
- Все тестовые файлы должны использовать формат `kebab-case`.

Предлагаемая структура:

1. `jsdoc-parser-syntax.test.ts`
2. `jsdoc-parser-fields.test.ts`
3. `jsdoc-parser-descriptions.test.ts`
4. `jsdoc-parser-validation.test.ts`
5. `jsdoc-parser-edge-cases.test.ts`

Разделение по группам обязательно: оно уменьшает размер файлов, упрощает навигацию и делает поддержку snapshot-ов предсказуемой.

## Правила содержимого тестов

- входной контракт задается inline строкой в тесте;
- результат проверяется snapshot-ом целиком как `DbcSchema`;
- один тест покрывает один сценарий;
- snapshots должны быть читаемыми и отражать полную форму `entries`, включая `issues` у каждой записи;
- если сценарий спорный и не закреплен в спецификации, его нельзя делать обязательным snapshot-кейсом без дополнительного уточнения.

## Группа 1. Синтаксис и обработка формата

Файл: `jsdoc-parser-syntax.test.ts`

### Кейс 1.1: Классический JSDoc со звездочками

- Given: вход начинается с `/**`, заканчивается `*/`, строки содержат ведущие `*`
- When: парсер обрабатывает текст
- Then: служебные символы JSDoc игнорируются, содержимое корректно парсится в `DbcSchema`, а каждая запись содержит свой `issues`
- Snapshot: должен содержать ожидаемую полную схему без артефактов `*`

### Кейс 1.2: Чистый текст без спецсимволов JSDoc

- Given: вход содержит только строки контракта без `/**`, `*/` и `*`
- When: парсер обрабатывает текст
- Then: результат эквивалентен JSDoc-варианту для того же контракта
- Snapshot: должен показывать ту же логическую структуру `DbcSchema` без корневого `issues`

### Кейс 1.3: Однострочный формат

- Given: контракт записан в одну строку, например `/** @purpose Simple description */`
- When: парсер обрабатывает текст
- Then: корректно выделяются `type` и `value`
- Snapshot: должен содержать одну запись `purpose` с пустым `issues`

## Группа 2. Извлечение полей

Файл: `jsdoc-parser-fields.test.ts`

### Кейс 2.1: Полный разбор `@param`

- Given: строка `@param {string} userId User identifier`
- When: парсер обрабатывает тег `param`
- Then: корректно извлекаются `type`, `dataType`, `specifier` и `value`
- Snapshot: должен содержать запись `param` со всеми ожидаемыми полями и `issues: []`

### Кейс 2.2: Опциональный параметр

- Given: строка `@param {number} [age] User age`
- When: парсер обрабатывает тег `param`
- Then: `specifier` извлекается без скобок, `optional` = `true`
- Snapshot: должен отражать `optional: true` и пустой `issues`

### Кейс 2.3: Тег `@see`

- Given: строка `@see {Interface#method} in ./path`
- When: парсер обрабатывает тег `see`
- Then: `specifier` извлекается из `{...}`, остаток попадает в `value`
- Snapshot: должен содержать запись `see` с корректным `specifier` и пустым `issues`

### Кейс 2.4: Неизвестный тег

- Given: строка `@author John Doe`
- When: тег не входит в список специальных контрактных тегов
- Then: он парсится по общему правилу как `type: author`, `value: John Doe`
- Snapshot: не должен содержать специальных полей `specifier` или `dataType`, если их не требует общий формат, и должен содержать пустой `issues`

### Кейс 2.5: `@returns` с типом данных

- Given: строка `@returns {boolean} True on success.`
- When: парсер обрабатывает тег `returns`
- Then: корректно извлекаются `type`, `dataType` и `value`
- Snapshot: должен подтверждать формат, показанный в спецификации, включая пустой `issues`

## Группа 3. Описания и многострочные значения

Файл: `jsdoc-parser-descriptions.test.ts`

### Кейс 3.1: Текст до первого тега

- Given: контракт начинается с нескольких строк текста, а первый тег идет позже
- When: парсер обрабатывает текст до первого `@`
- Then: этот текст собирается в одну запись `description`
- Snapshot: должен показывать первую запись типа `description` с пустым `issues`

### Кейс 3.2: Явный тег `@description`

- Given: в контракте есть тег `@description My info`
- When: парсер обрабатывает этот тег
- Then: он попадает в `entries` как запись типа `description`
- Snapshot: должен показывать `type: description` и `issues: []`

### Кейс 3.3: Многострочное значение у тега

- Given: значение тега занимает несколько строк, продолжения не начинаются с `@`
- When: парсер присоединяет строки к предыдущей записи
- Then: поле `value` собирается через `\n`
- Snapshot: должен явно фиксировать многострочное значение

### Кейс 3.4: Многострочный `@param`

- Given: описание параметра продолжается на следующей строке
- When: парсер обрабатывает продолжение
- Then: обе строки попадают в `value` одного `param`
- Snapshot: должен совпадать с примером из task-спецификации

## Группа 4. Валидация и ошибки

Файл: `jsdoc-parser-validation.test.ts`

### Кейс 4.1: Конфликт `@purpose` и `@see`

- Given: контракт содержит и `@purpose`, и `@see`
- When: выполняется валидация
- Then: `ERR_DBC_PURPOSE_CONFLICT` появляется в `issues` той записи, которая делает конфликт наблюдаемым
- Snapshot: должен содержать корректный код ошибки и ожидаемую строку внутри проблемной записи

### Кейс 4.2: Нарушение порядка тегов

- Given: `@returns` идет перед `@param`
- When: выполняется проверка порядка
- Then: `ERR_DBC_ORDER` появляется в `issues` записи, которая оказалась слишком поздней относительно уже прочитанных тегов
- Snapshot: должен отражать нарушение порядка по правилам из спецификации без корневого списка ошибок

### Кейс 4.3: Отсутствует имя параметра у `@param`

- Given: строка `@param {string}` (specifier отсутствует полностью)
- When: выполняется валидация `param`
- Then: `ERR_DBC_PARAM_NAME_MISSING` появляется в `issues` самой записи `param`
- Snapshot: должен содержать запись `param` и соответствующую ошибку внутри неё

### Кейс 4.4: Невалидный `@see`

- Given: тег `@see` без корректного `{specifier}`
- When: выполняется валидация `see`
- Then: `ERR_DBC_SEE_FORMAT_INVALID` появляется в `issues` самой записи `see`
- Snapshot: должен содержать ошибку формата `see` внутри этой записи

### Кейс 4.5: Невалидный контракт из спецификации

- Given: пример невалидного контракта из [task-спецификации](services/dbc/parser/dbc-parser.task.spec.md)
- When: парсер обрабатывает этот текст
- Then: итоговый snapshot должен соответствовать эталонной схеме из спецификации по `entries` и их локальным `issues`
- Snapshot: используется как главный регрессионный тест на согласованность реализации и ТЗ

## Группа 5. Краевые случаи

Файл: `jsdoc-parser-edge-cases.test.ts`

### Кейс 5.1: Пустой контракт

- Given: вход пуст или содержит только пробелы
- When: парсер обрабатывает текст
- Then: `entries` пуст
- Snapshot: должен содержать пустую `DbcSchema` без корневого `issues`

### Кейс 5.2: Экстремальные пробелы и табы

- Given: между тегом, типом и именем есть множественные пробелы или табы
- When: парсер обрабатывает текст
- Then: сущности выделяются корректно, лишние пробелы не ломают структуру
- Snapshot: должен подтверждать нормализованный результат

### Кейс 5.3: Неизвестные теги не участвуют в контрактной валидации

- Given: контракт содержит generic tag, который не входит в контрактные правила
- When: выполняется парсинг и валидация
- Then: тег должен быть распознан, но не должен сам по себе вызывать contract-specific issue
- Snapshot: должен показывать корректную запись без лишних `issues` в самой записи

### Кейс 5.4: Неопределенное поведение по malformed datatype

- Given: строка наподобие `@param {string description`
- When: парсер обрабатывает некорректный синтаксис типа
- Then: этот сценарий нельзя закреплять как обязательный эталонный snapshot, пока ожидаемое поведение не зафиксировано отдельно
- Статус: open question; не включать в обязательный регрессионный набор без дополнительного решения

## Контрольный список качества плана

- все обязательные правила валидации из спецификации покрыты;
- snapshots проверяют отсутствие корневого `issues` и наличие `issues` у каждой записи;
- коды ошибок совпадают со спецификацией:
  - `ERR_DBC_PURPOSE_CONFLICT`
  - `ERR_DBC_ORDER`
  - `ERR_DBC_PARAM_NAME_MISSING`
  - `ERR_DBC_SEE_FORMAT_INVALID`
- snapshots используются как проверяемый эталон результата, а не как автоматическое отражение текущего вывода;
- snapshot-файлы лежат только в отдельной папке `__tests__/snapshots`;
- входные данные задаются только inline в тестах;
- тесты разложены по нескольким `kebab-case.test.ts` файлам;
- спорные сценарии не попадают в обязательный snapshot-набор без явного решения по expected behavior.
- покрытие кода тестами должно стремиться к 100%.
- перед сдачей обязательно выполнены:
  - type-check командой `npm run type-check` (если в проекте нет скрипта `type-check`, его нужно добавить перед запуском);
  - coverage-прогон через обычный запуск тестов с флагом покрытия:
    - весь тестовый набор: `npm test -- --experimental-test-coverage`;
    - конкретный тест/набор тестов: `npx tsx --test --experimental-test-coverage <path-or-glob-to-tests>`;
  - в отчете обязательно указать итоговые проценты и перечислить непокрытые участки.
