## 1. Обзор Системы

### 1.1. Назначение системы

Система `dbc/parser` предназначена для парсинга и валидации текстовых **контрактов на написание кода (Design by Contract)**.

Ключевая задача — не просто извлечь метаданные, а проверить их на соответствие внутреннему стандарту контрактов. Система должна анализировать текстовый блок, преобразовывать его в структурированную модель данных и формировать список найденных нарушений правил контракта, привязанный к конкретным распознанным записям.

### 1.2. Архитектурный подход

Система должна быть спроектирована как модульный фреймворк:

1.  **Ядро:** Определяет универсальный интерфейс парсера `DbcParser` и общие структуры данных (`DbcSchema`, `DbcEntrySchema`, `DbcDbcEntryIssue`). Диагностика хранится на уровне записи, а не на уровне корневой схемы.
2.  **Реализации:** Для каждого синтаксиса контрактов создаётся отдельный пакет-реализация. Первой такой реализацией будет парсер для контрактов, использующих JSDoc-подобный синтаксис.

### 1.3. Ключевые принципы

- **Языковая независимость:** Данное ТЗ описывает логику и структуры, которые могут быть реализованы на любом языке программирования (Go, Python, Rust, JavaScript и т.д.).
- **Фокус на контракте:** Система валидирует не соответствие стандарту JSDoc, а соответствие внутренним правилам оформления контрактов, которые лишь используют JSDoc как транспорт.

## 3. Универсальная Схема Данных (Модель)

Это основа системы. Любая реализация парсера ДОЛЖНА возвращать данные в этом формате.

### 3.1. DbcSchema

Корневой объект, представляющий результат парсинга всего контракта.

| Поле      | Тип                              | Обязательность | Описание                                                                                                           |
| :-------- | :------------------------------- | :------------- | :----------------------------------------------------------------------------------------------------------------- |
| `entries` | Массив объектов `DbcEntrySchema` | Да             | Список всех успешно распознанных записей (тегов) из контракта. Каждая запись содержит собственный массив `issues`. |

### 3.2. DbcEntrySchema

Объект, представляющий одну распарсенную запись (тег) в контракте.

| Поле        | Тип                                | Обязательность | Описание                                                                                       |
| :---------- | :--------------------------------- | :------------- | :--------------------------------------------------------------------------------------------- |
| `type`      | Строка                             | Да             | Тип записи. Соответствует названию тега без `@` (например, `param`, `returns`).                |
| `specifier` | Строка                             | Нет            | Идентификатор, к которому относится запись. Обычно это имя переменной или сущности.            |
| `dataType`  | Строка                             | Нет            | Тип данных, указанный в фигурных скобках `{...}`. Скобки в значение не включаются.             |
| `optional`  | Логический                         | Нет            | Флаг, указывающий на опциональность (например, для `[specifier]`).                             |
| `value`     | Строка                             | Да             | Основное текстовое описание, следующее за типом и спецификатором.                              |
| `issues`    | Массив объектов `DbcDbcEntryIssue` | Да             | Список проблем, относящихся именно к этой записи. Если проблем нет, массив должен быть пустым. |

### 3.3. Правила Привязки `issues`

Модель ДОЛЖНА подчиняться следующим правилам:

1.  Поле `issues` существует у **каждой** записи `DbcEntrySchema`, включая `description` и любые generic-теги.
2.  Если у записи нет нарушений, `issues` ДОЛЖЕН быть пустым массивом `[]`, а не `null` и не отсутствующим полем.
3.  Любой объект `DbcDbcEntryIssue` ДОЛЖЕН храниться только внутри той записи, к которой он относится. Корневой `DbcSchema` не содержит собственного агрегирующего массива `issues`.
4.  Если правило валидации сравнивает несколько записей, объект `DbcDbcEntryIssue` ДОЛЖЕН привязываться к той записи, на которой нарушение становится наблюдаемым при чтении контракта слева направо.
5.  Поле `line`, если оно заполнено, ДОЛЖНО указывать на стартовую строку той записи, в `issues` которой хранится соответствующий issue.

### 3.4. DbcDbcEntryIssue

Объект, представляющий одну ошибку валидации, найденную в контракте.

| Поле   | Тип    | Обязательность | Описание                                                      |
| :----- | :----- | :------------- | :------------------------------------------------------------ |
| `code` | Строка | Да             | Уникальный код ошибки (например, `ERR_DBC_ORDER`).            |
| `line` | Число  | Нет            | Номер строки в исходном тексте, где была обнаружена проблема. |

### 3.5. DbcParser

Базовый интерфейс парсера контрактов. Любая реализация парсера ДОЛЖНА реализовывать этот интерфейс.

Интерфейс ДОЛЖЕН быть объявлен и экспортирован из `services/dbc/parser/dbc-parser.types.ts`.

**Сигнатура:**

`parse(inputContract: string): DbcSchema`

## 4. Спецификация Реализации: `dbc-jsdoc-parser`

Эта реализация парсит контракты, написанные с использованием JSDoc-подобного синтаксиса.

Реализация ДОЛЖНА быть представлена классом `DbcJsDocParser`, который реализует `DbcParser`.

### 4.1. Общие правила парсинга

1.  **Обработка строк:** Парсер должен обрабатывать входной текст построчно.
2.  **Определение тега:** Строка считается началом нового тега, если она (после удаления начальных пробелов, а также опционального ведущего символа `*`, если он присутствует) начинается с символа `@`. Входной текст может быть как в исходном JSDoc-виде (строки начинаются с `*`), так и уже предварительно очищенным от `*`.
3.  **Многострочные значения:** Если строка не является началом нового тега, её текст ДОЛЖЕН быть присоединён к полю `value` предыдущей обработанной записи. В качестве разделителя используется символ новой строки (`\n`).
4.  **Обработка `description`:** Весь текст, который находится до первого тега (`@`), должен быть собран в одну запись типа `description`. Если присутствует тег `@description`, его значение также обрабатывается как `description`.

### 4.2. Парсинг Тегов

#### 4.2.1. Основные теги контракта

Это теги с особыми правилами парсинга и валидации.

- `@purpose`, `@consumer`, `@invariant`, `@pre`, `@post`, `@sideEffect`, `@throws`, `@returns`
  - **Формат:** `@<type> <value>`
  - **Парсинг:** Первое слово — `type`, весь остаток строки — `value`.
- `@param`
  - **Формат:** `@param {<dataType>} [<specifier>] <value>`
  - **Парсинг:**
    - `dataType`: извлекается из `{...}`.
    - `specifier`: имя параметра. Если обрамлено в `[]`, поле `optional` устанавливается в `true`.
    - `value`: остаток строки.
- `@see`
  - **Формат:** `@see {<specifier>} <value>`
  - **Парсинг:**
    - `specifier`: извлекается из `{...}`.
    - `value`: остаток строки.

#### 4.2.2. Прочие теги

Любой другой тег, не входящий в список основных тегов контракта (например, `@author`, `@deprecated`, `@version`), должен парситься по общему правилу:

- **Формат:** `@<type> <value>`
- **Парсинг:** Первое слово после `@` становится `type`, остаток строки — `value`. К этим тегам правила валидации контракта не применяются.

### 4.3. Правила Валидации Контракта

Парсер ДОЛЖЕН проверять контракт на соответствие следующим правилам. При нарушении в массив `issues` соответствующей записи добавляется объект `DbcDbcEntryIssue`.

| Код ошибки                       | Описание                                                            | Логика проверки                                                                                                                                                                                                                                                          | К какой записи привязывать issue                                                      |
| :------------------------------- | :------------------------------------------------------------------ | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------ |
| **`ERR_DBC_PURPOSE_CONFLICT`**   | Нельзя одновременно использовать `@purpose` и `@see`.               | Проверить, что в массиве `entries` не возникает запись `purpose` после уже встреченного `see` или запись `see` после уже встреченного `purpose`.                                                                                                                         | К более поздней конфликтующей записи, которая делает конфликт наблюдаемым.            |
| **`ERR_DBC_ORDER`**              | Нарушен строгий порядок следования тегов контракта.                 | Проверить, что теги контракта в массиве `entries` следуют в порядке: `consumer`, `invariant`, `pre`, `param`, `throws`, `returns`, `post`, `sideEffect`. Запись нарушает правило, если её позиция в этом порядке меньше, чем у уже встреченного ранее контрактного тега. | К записи, которая встретилась слишком поздно относительно уже прочитанного контракта. |
| **`ERR_DBC_PARAM_NAME_MISSING`** | У тега `@param` отсутствует имя параметра (specifier).              | Проверить, что у каждой записи с `type: 'param'` заполнено поле `specifier`.                                                                                                                                                                                             | К конкретной записи `param` без `specifier`.                                          |
| **`ERR_DBC_SEE_FORMAT_INVALID`** | У тега `@see` отсутствует или некорректно указана ссылка в `{...}`. | Проверить, что у каждой записи с `type: 'see'` заполнено поле `specifier`.                                                                                                                                                                                               | К конкретной записи `see` без корректного `specifier`.                                |

## 5. Примеры Использования

### 5.1. Пример 1: Валидный контракт

**Входной текст:**

```text
/**
 * A very important function.
 * @consumer MainApp
 * @invariant Must be called after initialization.
 * @param {string} userId The ID of the user to process.
 * @param {object} [options] Optional settings.
 *   These settings can span multiple lines.
 * @returns {boolean} True on success.
 * @sideEffect Writes data to the database.
 */
```

**Ожидаемый результат (JSON):**

```json
{
  "entries": [
    {
      "type": "description",
      "value": "A very important function.",
      "issues": []
    },
    {
      "type": "consumer",
      "value": "MainApp",
      "issues": []
    },
    {
      "type": "invariant",
      "value": "Must be called after initialization.",
      "issues": []
    },
    {
      "type": "param",
      "dataType": "string",
      "specifier": "userId",
      "value": "The ID of the user to process.",
      "issues": []
    },
    {
      "type": "param",
      "dataType": "object",
      "specifier": "options",
      "optional": true,
      "value": "Optional settings.\nThese settings can span multiple lines.",
      "issues": []
    },
    {
      "type": "returns",
      "dataType": "boolean",
      "value": "True on success.",
      "issues": []
    },
    {
      "type": "sideEffect",
      "value": "Writes data to the database.",
      "issues": []
    }
  ]
}
```

## 6. Границы Системы (Что не входит в Scope)

- **Парсинг XML-подобных форматов:** Текущая архитектура не предназначена для парсинга форматов документации, основанных на XML (например, как в C#).
- **Восстановление иерархии объектов:** Система не должна пытаться восстановить вложенные связи между параметрами (например, `user` и `user.name`). Эта задача остаётся на стороне потребителя данных.
